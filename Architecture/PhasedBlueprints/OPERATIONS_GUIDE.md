# Phased Operations & Deployment Guide

This guide provides granular, step-by-step instructions for the setup, deployment, and maintenance of the DGLab ecosystem across its lifecycle stages.

---

## Stage 1: Development (The Lab)
**Goal**: Rapid iteration with high-fidelity local environment.

### Phase 1.1: Local Environment Setup
- [ ] **PHP 8.2+**: Ensure PHP is installed with `mbstring`, `pdo_sqlite`, `json`, `curl`, and `swoole` extensions.
- [ ] **Composer**: Run `composer install` to pull core framework dependencies.
- [ ] **SQLite Isolation**: The framework defaults to SQLite for local development. Run `php cli/migrate.php` to initialize the `database/database.sqlite` file.
- [ ] **Swoole for Nexus**: To test WebSockets, start the Nexus service: `php cli/nexus.php start`.

### Phase 1.2: Node-Free Asset Management
- [ ] **Bundle Generation**: Instead of `npm run build`, use the internal `AssetBundler`: `php cli/bundle.php`.
- [ ] **SuperPHP Live Preview**: Ensure `APP_DEBUG=true` in `.env` to enable the "Interpreted" mode for SuperPHP components, allowing immediate UI feedback.

### Phase 1.3: Verification & Isolation
- [ ] **Filesystem Isolation**: Use the built-in `TestCase` helpers to ensure tests run in `sys_get_temp_dir()`.
- [ ] **Continuous Testing**: Run `vendor/bin/phpunit` frequently to verify logic integrity.

## Stage 2: Production (The Citadel)
**Goal**: Security, scalability, and zero-downtime reliability.

### Phase 2.1: Infrastructure Provisioning (Provisioning & Security)
- [ ] **Server Selection**: Use Ubuntu 22.04+ LTS. Install `nginx`, `redis-server`, and `php-fpm`.
- [ ] **Redis Grid Configuration**:
    - Enable **AOF (Append Only File)** for durability: `appendonly yes`.
    - Configure **RDB Snapshots**: `save 900 1`, `save 300 10`, `save 60 10000`.
    - **Security**: Require a strong password (`requirepass <secret>`) and bind only to localhost or a private VPC IP.
- [ ] **Nginx Reverse Proxy**:
    - Configure Nginx to serve the `public/` directory for HTTP.
    - Implement **WSS Termination** for Nexus:
      ```nginx
      location /nexus {
          proxy_pass http://127.0.0.1:9501;
          proxy_http_version 1.1;
          proxy_set_header Upgrade $http_upgrade;
          proxy_set_header Connection "Upgrade";
          proxy_set_header Host $host;
          proxy_set_header X-Real-IP $remote_addr;
          proxy_set_header X-Forwarded-Proto $scheme;
          proxy_read_timeout 86400; # Prevent timeouts for long-lived WS connections
      }
      ```

### Phase 2.2: Zero-Downtime Deployment (Atomic Deployments)
- [ ] **Deployer Integration**: Use the provided `deploy.php` recipe.
- [ ] **Deployment Hook for Nexus**:
  ```php
  # deploy.php snippet
  task('deploy:restart_nexus', function () {
      run('php {{release_path}}/cli/nexus.php restart');
  })->desc('Restart Nexus WebSocket server');

  after('deploy:symlink', 'deploy:restart_nexus');
  ```
- [ ] **Process Management**: It is recommended to use **Supervisord** to monitor the `nexus.php` process.
  ```ini
  [program:nexus]
  command=php /var/www/current/cli/nexus.php start
  autostart=true
  autorestart=true
  user=www-data
  ```

### Phase 2.3: Security Hardening (Keys & Tenancy)
- [ ] **JWT Key Rotation**: Use `php cli/auth.php rotate-keys` to generate new RSA pairs for the AuthService.
- [ ] **Tenant Isolation**: Verify `X-Tenant-Id` headers are strictly enforced at the Nginx or Middleware level.

## Stage 3: Maintenance (The Forge)
**Goal**: Ongoing health, updates, and structural alignment.

### Phase 3.1: Blueprint-Sync Audits (Architectural Health)
- [ ] **Architectural Integrity**: Run `python3 self_created_tools/validate_blueprint.py` weekly to ensure the physical codebase hasn't drifted from `Architecture/codebase_blueprint.json`.
- [ ] **Phased Progression**: Update `Architecture/PhasedBlueprints/MASTER_BLUEPRINT.json` as new phases are completed.

### Phase 3.2: Data Lifecycle & Migrations (Stability)
- [ ] **Migration Safety**: Always perform `php cli/migrate.php --dry-run` before applying production schema changes.
- [ ] **Cleanup Service**: Ensure the `DownloadService` cleanup cron job is running to reclaim storage from expired tokens.

### Phase 3.3: Security Patching & Observability (Long-term)
- [ ] **Dependency Audit**: Run `composer audit` monthly to identify vulnerable packages.
- [ ] **Pulse Monitoring**: Use the CMS Studio **Pulse App** to monitor server load, WebSocket connection counts, and AI orchestration latency.
