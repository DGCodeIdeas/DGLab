# ADR-001: Dependency Injection Container Design

## Status
Accepted

## Context
The Sovereign Stack requires a dependency injection (DI) container as the foundational wiring mechanism. Every core component, from the Router to the Event Dispatcher, must be resolved through this container. Key requirements from [CORE-02](/ApprovedBlueprints/Core/CORE-02.md):

- **PSR-11 compliance** for interoperability with ecosystem tools
- **Sub-1ms resolution time** for pre-compiled service maps to avoid bottlenecking request throughput
- **PHP 8.3 Attribute support** for explicit service configuration (a core language feature differentiator)
- **Recursive reflection-based auto-wiring** to reduce manual wiring for simple dependency chains
- Support for singletons, factory bindings, and parameterized instantiation

## Decision
Adopt a **two-mode DI container architecture**:

1. **Development Mode**: Reflection-based auto-wiring using `ReflectionClass` to resolve constructor dependencies recursively. Services are registered lazily and resolved on-demand with full error messages for missing bindings.

2. **Production Mode**: Compiled service map - a flat PHP array generated by a `CompilerPass` analyzing the dependency graph ahead of time. This eliminates reflection overhead entirely for production deployments.

### Core Interface
```php
interface ContainerInterface extends \Psr\Container\ContainerInterface {
    public function bind(string $id, mixed $concrete = null, bool $singleton = false): void;
    public function make(string $id, array $parameters = []): mixed;
}
```

### Key Behaviors
- `bind($id, $concrete, true)` registers a singleton (same instance on every resolution)
- `make($id, $parameters)` allows parameter overrides for ad-hoc resolution
- Resolution chain: compiled map > registered bindings > reflection auto-wiring

## Rationale
- **Performance**: The compiled mode (flat PHP array) achieves the sub-0.5ms resolution target for 3-level deep trees, exceeding the PSR-11 baseline while maintaining compliance
- **Developer Experience**: Reflection auto-wiring eliminates boilerplate for 80% of services; only edge cases (interfaces, primitives) need explicit `bind()` calls
- **Future-Proofing**: PHP 8.3 Attributes can be added later for metadata-driven wiring without changing the core interface
- **Separation of Concerns**: The `ContainerInterface` intentionally extends PSR-11 rather than customizing it, allowing drop-in replacement with other PSR-11 containers

## Consequences
### Positive
- All 20 core blueprints can be wired without circular reference issues (the `ServiceProvider` system in [CORE-17](/ApprovedBlueprints/Core/CORE-17.md) enforces `register()` before `boot()` ordering)
- Downstream blueprints (Hub/Spoke) can bind their own services without modifying core code
- Compiled service maps enable Opcache optimization across requests

### Negative
- Reflection auto-wiring in development adds ~2-3ms per unresolved service (mitigated by compiled maps in production)
- Container cannot auto-wire scalar parameters or interface bindings without explicit registration
- Compiled maps must be regenerated when bindings change (handled by `s-forge cache:clear`)

## Alternatives Considered
1. **PHP-DI / Laravel-style Container** - Feature-rich but introduces external dependency; violates the core principle of minimizing third-party coupling. Rejected for portability.
2. **Pure Service Locator pattern** - Simpler but violates PSR-11 and makes it impossible to swap implementations without changing consumer code. Rejected for maintainability.
3. **Ahead-of-Time Code Generation** (no runtime reflection) - Would be fastest but requires a build step; adds complexity to development workflow. Deferred as potential future optimization.
